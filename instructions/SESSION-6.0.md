## What Changed in Session 6.0

Session 5.3-frontend completed the user account status management feature with frontend status display, toggle functionality, and OAuth disabled account error handling. Session 6.0 introduces comprehensive backup management for the Laravel application, installing the Spatie Laravel Backup package to handle automated database and storage backups, publishing the backup configuration file that specifies what to backup (storage directory and database), configuring source files to include and exclude (vendor and node_modules excluded), setting backup destination to local disk with ZIP compression, adding database dump SSL verification skip option to the MySQL connection configuration to prevent certificate errors during backup, creating a CreateBackupJob that dispatches the backup:run Artisan command to the queue for asynchronous processing, implementing a BackupController with four endpoints for listing backups with metadata (filename, size, date), creating new backups via queued jobs, downloading backup files, and deleting backup files, registering backup routes under the admin middleware prefix alongside user management routes, scheduling automatic daily backups at 2:00 AM via Laravel's task scheduler, and updating the supervisord configuration to run the Laravel scheduler continuously as a background process.

| Area | Session 5.3-frontend | Session 6.0 |
|---|---|---|
| Package dependencies | No backup package | spatie/laravel-backup:^10.3 installed via Composer |
| Backup configuration | No backup config | config/backup.php published with storage and database sources, local disk destination |
| Database dump options | No SSL handling | MySQL connection configured with --skip-ssl-verify-server-cert flag |
| Backup job | No backup jobs | CreateBackupJob queues backup:run command for async execution |
| Backup API endpoints | No backup endpoints | Four endpoints: GET list, POST create, GET download, DELETE delete |
| API routes | User management only | Backup routes added under /manage/backups prefix with admin middleware |
| Scheduled tasks | No scheduled tasks | backup:run scheduled daily at 2:00 AM |
| Supervisor services | Octane server and queue worker only | Added laravel-schedule program to run schedule:work continuously |

`laravel-app/composer.json` existed previously and was modified by running `composer require spatie/laravel-backup:^10.3`. `laravel-app/composer.lock` was automatically updated by Composer. `laravel-app/config/backup.php` was generated by running `php artisan vendor:publish --provider="Spatie\Backup\BackupServiceProvider"` without further edits. `laravel-app/config/database.php` existed previously and was edited manually to add MySQL dump options. `laravel-app/app/Jobs/CreateBackupJob.php` was generated by running `php artisan make:job CreateBackupJob` and then manually edited to implement the handle method. `laravel-app/app/Http/Controllers/API/BackupController.php` was generated by running `php artisan make:controller API/BackupController` and then manually edited to implement all backup management methods. `laravel-app/routes/api.php` existed previously and was edited manually to register backup routes. `laravel-app/routes/console.php` existed previously and was edited manually to schedule the daily backup. `docker/laravel/supervisord.development.conf` existed previously and was edited manually to add the Laravel scheduler worker.

### `laravel-app/app/Http/Controllers/API/BackupController.php`

> **Generated by command, then manually edited** — scaffold the backup controller, then implement methods for listing, creating, downloading, and deleting backups.

```bash
php artisan make:controller API/BackupController
```

```php
<?php

namespace App\Http\Controllers\API;

use App\Jobs\CreateBackupJob;
use Exception;
use App\Http\Controllers\Controller;
use Illuminate\Support\Facades\Storage;

class BackupController extends Controller
{
    private $disk;

    private $folder;

    public function __construct()
    {
        $this->disk = Storage::disk(config('backup.backup.destination.disks')[0]);

        $this->folder = config('backup.backup.name');
    }

    public function createBackup()
    {
        CreateBackupJob::dispatch();
        return response([
            'message' => 'Backup process started.'
        ], 202);
    }

    public function getBackups()
    {
        $backups = [];
        $total_backups = 0;
        $total_size = 0;

        try {
            $files = $this->disk->files($this->folder);

            foreach ($files as $file) {
                $filename = basename($file);
                $extension = pathinfo($filename, PATHINFO_EXTENSION);
                if ($extension !== 'zip') {
                    continue;
                }
                $size = $this->disk->size($file);
                $size_human = $this->formatBytes($size);
                $date = $this->disk->lastModified($file);
                $date_human = date('Y-m-d H:i:s', $date);

                $total_backups++;
                $total_size += $size;

                $backups[] = [
                    'filename' => $filename,
                    'size' => $size,
                    'date' => $date,
                    'size_human' => $size_human,
                    'date_human' => $date_human,
                ];
            }
            $backups = array_reverse($backups);
        } catch (Exception $e) {
            return response([
                'message' => 'Failed to retrieve backups'
            ], 500);
        }

        return response([
            'backups' => $backups,
            'total_backups' => $total_backups,
            'total_size' => $total_size,
            'total_size_human' => $this->formatBytes($total_size),
        ], 200);
    }

    public function downloadBackup($filename)
    {
        $path = $this->folder . '/' . $filename;

        if (!$this->disk->exists($path)) {
            return response([
                'message' => 'Backup file not found.'
            ], 404);
        }

        try {
            return $this->disk->download($path);
        } catch (Exception $e) {
            return response([
                'message' => 'Failed to download backup file.'
            ], 500);
        }
    }

    public function deleteBackup($filename)
    {
        $path = $this->folder . '/' . $filename;

        if (!$this->disk->exists($path)) {
            return response([
                'message' => 'Backup file not found.'
            ], 404);
        }

        try {
            $this->disk->delete($path);
        } catch (Exception $e) {
            return response([
                'message' => 'Failed to delete backup file.',
            ], 500);
        }

        return response([
            'message' => 'Backup deleted successfully.'
        ], 200);
    }

    private function formatBytes($bytes, $precision = 2)
    {
        $units = ['B', 'KB', 'MB', 'GB', 'TB'];
        $bytes = max($bytes, 0);
        $pow = floor(($bytes ? log($bytes) : 0) / log(1024));
        $pow = min($pow, count($units) - 1);
        $bytes /= pow(1024, $pow);
        return round($bytes, $precision) . ' ' . $units[$pow];
    }
}
```

## How Each File Works

### Package installation and backup configuration

The Spatie Laravel Backup package provides a comprehensive solution for backing up Laravel applications, including database dumps and file backups. Running `composer require spatie/laravel-backup:^10.3` installs the package and its dependencies, including spatie/db-dumper for database exports and spatie/temporary-directory for temporary file handling. Publishing the package's configuration via `php artisan vendor:publish` creates `config/backup.php`, which defines backup behavior: the `source.files.include` array specifies `storage_path()` as the directory to backup, `source.files.exclude` prevents vendor and node_modules from being included, `source.databases` points to the default database connection from `DB_CONNECTION`, `destination.disks` targets the local filesystem driver, and `destination.compression_level` sets ZIP compression to maximum strength (9). The published config also includes cleanup strategies that retain all backups for 7 days, then keep daily backups for 16 days, weekly backups for 8 weeks, monthly backups for 4 months, and yearly backups for 2 years, ensuring backup storage remains manageable.

---

### Database dump options and SSL certificate handling

The Spatie Backup package uses spatie/db-dumper to export database contents, which internally invokes the `mysqldump` command-line utility for MySQL connections. When the MySQL server is configured with SSL, mysqldump may fail if the server's certificate cannot be verified, returning a certificate verification error. Adding the `dump` array to the `mysql` connection configuration in `config/database.php` allows passing custom options to mysqldump. The `add_extra_option` key with value `--skip-ssl-verify-server-cert` instructs mysqldump to skip server certificate validation, preventing SSL-related backup failures in development or containerized environments where certificate chains may not be fully configured. This option does not affect Laravel's runtime database queries, only the backup process. Production environments with valid SSL certificates may omit this option for stricter security.

---

### Backup job — queued backup execution

The `CreateBackupJob` implements the `ShouldQueue` interface to ensure backup creation does not block HTTP responses, as database dumps and file archiving can take several seconds or longer depending on data volume. The job's `handle` method calls `Artisan::call('backup:run')`, which triggers the Spatie Backup package's backup process: the package reads the configuration from `config/backup.php`, creates a temporary directory for intermediate files, exports the database using spatie/db-dumper (which runs mysqldump with the configured options), copies all files from `storage_path()` excluding vendor and node_modules, creates a timestamped ZIP archive combining the database dump and files, moves the archive to the configured disk (local storage in this case), cleans up the temporary directory, and optionally sends notifications on success or failure. Dispatching this job from the controller allows the API to immediately return a 202 Accepted response while the backup runs asynchronously in the background queue worker.

---

### Backup controller — API endpoints for backup management

The `BackupController` provides four admin-only endpoints for managing application backups. The constructor initializes `$this->disk` from the first disk in `config('backup.backup.destination.disks')` and sets `$this->folder` to `config('backup.backup.name')`, which defaults to the application name, establishing the storage location for backup files. The `getBackups` method lists all ZIP files in the backup folder, extracting metadata for each: filename, file size in bytes, human-readable size via `formatBytes`, last modified timestamp, and human-readable date, returning an array of backups sorted newest-first plus aggregate totals. The `createBackup` method dispatches `CreateBackupJob` and immediately returns a 202 Accepted status, allowing the client to poll the list endpoint to detect when the backup completes. The `downloadBackup` method accepts a filename parameter, validates the file exists in the backup folder, and streams the file to the client using Laravel's `download` response, setting appropriate headers for browser file downloads. The `deleteBackup` method validates file existence and calls `$this->disk->delete($path)` to remove the backup file, useful for manually cleaning up old or failed backups. All methods wrap disk operations in try-catch blocks to return 500 errors if filesystem operations fail.

---

### API routes — backup endpoints registration

The backup routes are registered under `Route::prefix('backups')` within the existing `Route::middleware('admin')->prefix('manage')` group, placing them alongside user management routes at `/manage/backups/*`. This prefix structure ensures all backup endpoints require authentication via `auth:sanctum` and admin role verification via the `admin` middleware, preventing non-admin users from accessing sensitive backup operations. The four routes map HTTP verbs to controller methods: `GET /` lists all backups, `POST /create` queues a new backup job, `GET /download/{filename}` downloads a specific backup file by name, and `DELETE /delete/{filename}` removes a backup file. The download and delete routes use route parameter binding to pass the filename from the URL to the controller methods, allowing the client to target specific backups without additional request body parsing.

---

### Console schedule — automated daily backups

Laravel's task scheduler allows defining recurring commands in `routes/console.php` that run automatically when the scheduler is active. The single line `Schedule::command('backup:run')->dailyAt('02:00')` registers the backup command to execute every day at 2:00 AM server time, a common choice for scheduled backups as it typically falls outside peak usage hours, minimizing resource contention with user traffic. The scheduler uses the `schedule:work` command to check for due tasks every minute without requiring cron configuration. When the scheduled time arrives, the scheduler invokes `php artisan backup:run` directly, not via the queue, creating the backup synchronously. If a backup is already running from a queued job, the Spatie Backup package's locking mechanism prevents duplicate execution. The scheduled backup complements the manual API-triggered backups, ensuring daily snapshots even if admins forget to manually create them.

---

### Supervisor configuration — scheduled task worker

Laravel's task scheduler requires a continuously running process that checks for due tasks every minute, traditionally accomplished by a cron entry invoking `schedule:run` every minute. In containerized environments using supervisord, the `schedule:work` command provides an alternative: it runs indefinitely, checking the schedule internally every 60 seconds without requiring external cron. Adding the `[program:laravel-schedule]` section to `supervisord.development.conf` starts the scheduler as a supervised background process alongside the Octane server and queue worker. The `autostart=true` directive launches the scheduler when supervisord starts (i.e., when the Docker container boots), `autorestart=true` restarts the process if it crashes, ensuring the scheduler remains active. The `directory=/var/www/html` setting ensures the command runs from the Laravel application root. This configuration guarantees the scheduled daily backup executes reliably as long as the container is running, without manual intervention or external cron configuration.

## Common Commands

```bash
# Install the Spatie Laravel Backup package
composer require spatie/laravel-backup:^10.3

# Publish the backup package configuration
php artisan vendor:publish --provider="Spatie\Backup\BackupServiceProvider"

# Generate the backup job class
php artisan make:job CreateBackupJob

# Generate the backup controller
php artisan make:controller API/BackupController

# Manually run a backup (for testing)
php artisan backup:run

# List existing backups
php artisan backup:list

# Clean up old backups based on retention rules
php artisan backup:clean

# Check backup health
php artisan backup:monitor

# Restart Docker containers to apply supervisord changes
docker compose restart

# View scheduler logs in Docker
docker compose logs -f laravel

# View queue worker logs
docker compose exec laravel tail -f storage/logs/laravel.log
```
